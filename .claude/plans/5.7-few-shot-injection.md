# Plan: 5.7 — Few-Shot Injection

**Branch:** `feature/5.7-few-shot-injection`
**Goal:** Improve classification accuracy by injecting keyword-matched few-shot examples
from training data (and optionally `classifications.jsonl`) into the Ollama prompt.

---

## Context

The classifier currently sends Ollama a system prompt containing only the taxonomy
(68 subcategory→category mappings) plus the expense's `item`/`value`/`date`. No examples,
no history. The model guesses purely from pre-trained knowledge + the taxonomy list.

5.7 adds the first layer of the retrieval cascade (§ `data/classification/retrieval-strategy.md`):
keyword-matched few-shot examples injected as user/assistant conversation turns before
the real query. This gives the model concrete examples of how similar expenses were
previously classified.

**Token budget:** ~462 tokens baseline → ~660 with 5 examples (~20 tokens each). 2% of
32K context window. Not a bottleneck. [ref:retrieval-token-budget]

**Data sources:**
- `training_data_complete.json` (694 labeled entries, always available) [ref:training-data-schema]
- `classifications.jsonl` (runtime feedback, grows over time, may not exist) [ref:training-data-schema]

## Implementation Phases

### Phase 1: Example Selection Engine (`internal/classifier/examples.go`)

**New file.** Pure functions, no I/O (I/O is in the loader, Phase 2).

1. **Define types:**

```go
// Example is a labeled expense used as a few-shot prompt example.
type Example struct {
    Item        string
    Value       float64
    Date        string  // DD/MM or YYYY-MM-DD (normalized on load)
    Subcategory string
    Category    string
    Source      ExampleSource // Training, Confirmed, Corrected
}

type ExampleSource int
const (
    SourceTraining  ExampleSource = iota
    SourceConfirmed
    SourceCorrected
)
```

2. **Keyword matching function:**

```go
func SelectExamples(item string, pool []Example, keywords KeywordIndex, topK int) []Example
```

- Tokenize input `item` (lowercase, remove special chars, split whitespace, min length 2)
  — use same preprocessing rules as `algorithm_parameters.json` → `preprocessing`
- Look up each token in `KeywordIndex` (loaded from `feature_dictionary_enhanced.json` → `lexical_features.keywords`)
- Collect candidate subcategories with specificity scores
- For high-specificity matches (≥0.7): select examples from the dominant subcategory
- For ambiguous matches (<0.7): select examples from top-2 candidate subcategories (diversity)
- Sort: corrected > training > confirmed within matched subcategories
- Return top-K examples (default: 5)
- If no keyword matches: return empty slice (taxonomy-only fallback)

3. **KeywordIndex type:**

```go
type KeywordEntry struct {
    DominantSubcategory string
    Specificity         float64
    Subcategories       []string
}

type KeywordIndex map[string]KeywordEntry
```

**Existing code to reuse:**
- Preprocessing rules from `algorithm_parameters.json` (tracked file)
- Keyword data from `feature_dictionary_enhanced.json` → `lexical_features.keywords`

**Unit tests:** `internal/classifier/examples_test.go` — table-driven, using testify (`assert`/`require`):
- Input with high-specificity keyword → examples from dominant subcategory
- Input with ambiguous keyword → diverse examples from multiple subcategories
- Input with no keyword match → empty slice
- Corrected entries prioritized over training/confirmed
- TopK limiting works correctly
- Tokenization: accents preserved, special chars removed, min word length 2

### Phase 2: Data Loading (`internal/classifier/loader.go`)

**New file.** Handles I/O for both data sources.

1. **Load training data:**

```go
func LoadTrainingExamples(dataDir string) ([]Example, error)
```

- Reads `training_data_complete.json`
- Converts each entry to `Example` with `Source: SourceTraining`
- Date normalization: training data uses `YYYY-MM-DD`, convert to `DD/MM` for prompt consistency
- Returns `nil, nil` if file doesn't exist (graceful degradation, not an error)

2. **Load feedback examples:**

```go
func LoadFeedbackExamples(path string) ([]Example, error)
```

- Reads `classifications.jsonl` line by line
- Filters: skip `status == "manual"` (no model prediction to learn from)
- Maps `status == "confirmed"` → `SourceConfirmed`, `status == "corrected"` → `SourceCorrected`
- For corrected entries: use `actual_subcategory`/`actual_category` (the correction, not the prediction)
- Returns `nil, nil` if file doesn't exist (cold start)

3. **Load keyword index:**

```go
func LoadKeywordIndex(dataDir string) (KeywordIndex, error)
```

- Reads `feature_dictionary_enhanced.json` → `lexical_features.keywords`
- Extracts: `dominant_subcategory`, `specificity`, `subcategories` per keyword
- Returns error if file doesn't exist (keywords are required, unlike examples)

4. **Merge function:**

```go
func MergeExamplePools(training, feedback []Example) []Example
```

- Concatenates both pools
- Deduplication: if same `item` appears in both training and feedback, keep the feedback version (more recent/corrected)

**Unit tests:** `internal/classifier/loader_test.go` — table-driven, using testify (`assert`/`require`):
- Load training data from test fixture JSON
- Load feedback from test fixture JSONL (confirmed + corrected + manual entries)
- Manual entries filtered out
- Corrected entries use actual_* fields
- Missing file returns nil, nil (not error)
- Merge deduplicates by item

**Existing code to reuse:**
- `feedback.Entry` struct in `internal/feedback/feedback.go` — for parsing classifications.jsonl
- `LoadTaxonomy` in `classifier.go` already reads `feature_dictionary_enhanced.json` — extend or
  add a sibling function for keywords

### Phase 3: Prompt Construction — Few-Shot Injection

**Modify:** `expense-reporter/internal/classifier/classifier.go`

1. **Extract prompt to template file** (`data/classification/prompts/classify.tmpl`):

Since there's no `text/template` precedent in the codebase, use a simpler approach:
keep the `strings.Builder` pattern but move the static text portions to a constant or
a loadable text file. The dynamic parts (taxonomy, examples, topN) remain in Go code.

**Decision:** Use a Go constant (not a file) for the static prompt text. This avoids
adding a new file dependency that must be located at runtime. The prompt is small enough
that tracking changes via `git diff` on the Go file is sufficient. If prompt iteration
velocity increases later, extract to a file then.

> **Update note:** If you disagree with this decision, discuss with the user before
> implementing a file-based template. The user explicitly asked for prompt extraction
> to a trackable file — see discussion context. A middle ground: extract to a Go
> `embed.FS` file (compile-time embedded, tracked in git, no runtime path resolution).

2. **Modify `buildRequest`** to accept examples:

```go
func buildRequest(item string, value float64, date string, taxonomy Taxonomy, examples []Example, cfg Config) ([]byte, error)
```

- System prompt stays the same (taxonomy + instructions)
- Add user/assistant message pairs BEFORE the real user message:

```
Messages: [
  {role: "system", content: <system prompt>},
  // few-shot examples (injected)
  {role: "user", content: "item: Uber Centro\nvalue: 15.90\ndate: 05/03"},
  {role: "assistant", content: "{\"results\": [{\"subcategory\": \"Uber\", \"category\": \"Transporte\", \"confidence\": 0.95}]}"},
  // ... more examples ...
  // real query (last)
  {role: "user", content: "item: <actual item>\nvalue: <actual value>\ndate: <actual date>"},
]
```

- Each example becomes a user message (same format as real input) + assistant message
  (JSON matching the response schema, single result with confidence 0.95)
- Examples format helper: `func formatExampleMessages(examples []Example) []ollamaMessage`

3. **Modify `Classify`** to orchestrate example loading + selection:

```go
func Classify(item string, value float64, date string, taxonomy Taxonomy, cfg Config) ([]Result, error)
```

- Signature stays the same (backward compatible)
- Internally: load examples + keywords from `cfg.DataDir`, select via `SelectExamples`, pass to `buildRequest`
- Note: `cfg.DataDir` is already in the Config struct but currently unused at runtime.
  The commands pass it via CLI flags to `LoadTaxonomy` separately. Wire it properly:
  commands should set `cfg.DataDir` and let `Classify` handle all data loading internally.

4. **Verbose output** via existing `--verbose` flag:

- The `--verbose` flag already exists in `root.go`, sets `logger.SetDebug(verbose)`
- In `Classify`, after example selection, log:
  ```
  logger.Debug("few-shot: %d examples matched via keywords %v → %s (%s)", ...)
  ```
- This gives acceptance tests a deterministic string to assert on
- Check how `logger.Debug` works — it may need to exist or be added to the logger package

**Files to modify:**
- `internal/classifier/classifier.go` — buildRequest signature, Classify orchestration
- `cmd/expense-reporter/cmd/auto.go` — set `cfg.DataDir` instead of passing dataDir separately
- `cmd/expense-reporter/cmd/batch_auto.go` — same
- `cmd/expense-reporter/cmd/classify.go` — same
- `internal/logger/logger.go` — add `Debug` function if not present

**Unit tests:** `internal/classifier/classifier_test.go` — extend existing:
- buildRequest with examples produces correct message array
- buildRequest with empty examples produces same output as before (backward compat)
- formatExampleMessages produces correct user/assistant pairs
