## What changed

### `internal/taxonomy` (new pure helpers, `path.go`)
- `BuildPathMap` / `PathEnum` — the 112 `Type/Category/Subcategory` enum strings.
- `PathMap.Split` — **reverse-map lookup**, never a parse on `/`. Critical: 5 subs and
  2 categories contain `/` (`Uber/Taxi`, `Alimentação / Limpeza`), so splitting is wrong.
- `PathMap.PathFor` — forward, map-validated (triple → canonical enum string).
- `ResolveLeaf(sheets, sub, typeHint)` — bare-name → `(type, category)`; unique leaves
  resolve from the name, ambiguous ones need the hint else `ErrLeafAmbiguous`.
- `TypesForLeaf`, `CategoryForLeaf` — for `add`/`correct` messages & best-effort category.
- Tested against a **committed synthetic fixture** carrying the slash-in-name and
  cross-type hazards (the real taxonomy is gitignored). Sanity-checked out-of-band
  against the real 112-leaf tree (all round-trip; 5 ambiguous leaves error without hint).

### `internal/classifier`
- `Result` gains `Type`. `Taxonomy` type + `LoadTaxonomy` (feature-dict map) **deleted**.
- `Classify(item, value, date, sheets []taxonomy.ExpenseType, cfg)` — 3-level tree in
  the prompt; structured-output schema is an atomic `path` enum per candidate; response
  split back via `PathMap.Split`; off-enum paths dropped (never half-populated).
- Few-shot examples resolve their canonical path via `ResolveLeaf`+`PathFor` and are
  **dropped when unresolvable** — an example can never teach an off-enum answer.
  `Example.TypeHint` comes from the training `source` sheet-name (disambiguates the few
  cross-type leaves only). This was the advisor's key correction: building example paths
  from the training-file category string would re-introduce the category-disagreement bug.

### Commands (all 5 callers updated — build stays green)
- `auto` / `batch-auto`: write `Result.Type` directly; `resolveExpenseType` +
  `loadTypeIndex` deleted (2 obsolete command-layer tests removed; `LookupType` capability
  still covered in `taxonomy/lookup_test.go`).
- `add`: full path via `resolveFullPath` — unambiguous leaves resolve from the name; the
  ambiguous ones use a `--type`/interactive-prompt/non-interactive-hard-error hybrid
  (never a silent guess — matters for unattended runs). Made I/O-injectable for testing.
- `correct`: category via `CategoryForLeaf` (feedback-only, best-effort).
- `classify`/`auto`/`batch-auto`: default model `my-classifier-q3` → `my-classifier-qcoder`
  (the tier validated for the 112-path enum; CLAUDE.md already marks it required for 5.7+).
  Acceptance unaffected — the harness passes `--model` from the fixture explicitly.

## Decisions & deviations from plan

- **`SplitPath` is a method on `PathMap`, not a free function** (plan §4.2 wrote it free).
  Advisor-driven: the inverse genuinely needs the map because names contain `/`.
- **`§4.6` acceptance "type-less==0 on the model path" was NOT added as a CI assertion.**
  The only model-path acceptance step (batch-auto Step 1) is Ollama-gated and deliberately
  structural-only because it is non-deterministic; enum-honoring is model-dependent
  (validated on qcoder). That zero-fallback check belongs to the **WS-D real-data gate**,
  run manually with qcoder + real data — not CI. Deterministic routing is already proven
  by the apply→generate-workbook cycle.
